## Rule
Any `element.innerHTML = ...` assignment emits a warning. The warning is a prompt to review: is the right-hand side user-originated or fetched-from-API data? If yes, the assignment is an XSS vector and must be replaced with `textContent` (for plain strings) or a sanitized-HTML approach. If the right-hand side is a static string literal in source, the warning can be safely acknowledged.

## Why (rationale)
`innerHTML` parses the right-hand side as HTML, so any `<script>` tags, event-handler attributes, or `javascript:` URLs become live code. When the string comes from `fetch()` / URL parameters / form input, an attacker who controls that string controls the DOM. `textContent` is the safe alternative for displaying text; sanitization libraries (or a vetted allowlist parser) are the safe alternative for displaying HTML.

Warn (not error) because some legitimate static-string uses exist and blocking them would force awkward workarounds. The warning forces a per-use review.

## Pass example (static — acknowledged)
```js
element.innerHTML = '<span class="badge">New</span>';  // WARN, review OK: static literal, no user input
```

## Pass example (safe alternative)
```js
element.textContent = userName;  // No innerHTML, no XSS vector
```

## Fail (warning, actual risk) example
```js
const data = await fetch(apiUrl).then(r => r.text());
element.innerHTML = data;  // WARN — and actually unsafe, must be fixed
```

## Fix guidance
- For plain text: use `textContent`.
- For structured content from your own trusted sources (JSON→HTML render): build DOM nodes programmatically with `createElement` + `textContent`, not string concatenation.
- For rich HTML from unvetted sources: use a sanitizer (DOMPurify) or a strict allowlist parser. Do NOT write a custom regex-strip; they all fail.
- Static-string uses of innerHTML can stay, but commenting "// SEC-004 reviewed: static literal" makes the review state clear.
