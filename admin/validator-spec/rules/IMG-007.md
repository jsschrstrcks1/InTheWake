## Rule
The hero image src must end in `.webp`. No JPEGs, no PNGs, no GIFs — regardless of source format during editing.

## Why (rationale)
WebP is ~77% smaller than equivalent JPEG at the same visual quality. On a cruise-planning site where every port page has a large hero image, serving JPEG costs mobile users real bandwidth and LCP. The site standard has been WebP-only for new images since the WebP rollout. Standards docs and the validator agree.

Note: `logo_wake.png` is a documented exception elsewhere (transparency needs) but does NOT appear as a hero.

## Pass example
```html
<section class="port-hero">
  <img src="/ports/img/nassau/nassau-hero.webp" alt="..."/>
</section>
```

## Fail example
```html
<section class="port-hero">
  <img src="/ports/img/nassau/nassau-hero.jpg" alt="..."/>
</section>
```
Validator emits: `Hero image must be in webp format`.

## Fix guidance
Convert the hero source to WebP. Keep original JPEG in a non-deployed directory for re-conversion if needed. Do NOT rename a .jpg to .webp — the validator only checks extension, but serving mis-labeled content triggers downstream problems.
