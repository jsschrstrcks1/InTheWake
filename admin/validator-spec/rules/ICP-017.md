## Rule
The `<meta name="content-protocol">` tag must declare one of three accepted values during the ICP-Lite v1.4 → ICP-2 v2.1 migration:
- `ICP-Lite v1.4`
- `ICP-2`
- `ICP-2 v2.1`

Any other value (including empty, misspelled, or older versions like `ICP-Lite v1.0`) fails.

## Why (rationale)
The protocol version declares which rule set the page claims to conform to. Validators use it to apply the right checks. A page claiming `ICP-Lite v1.0` would fail today's checks anyway — forcing declaration to one of the current three is cleanup for a half-migrated fleet.

Once the migration completes and all pages are on ICP-2, the accepted set will shrink to just `ICP-2` / `ICP-2 v2.1`.

## Pass example
```html
<meta name="content-protocol" content="ICP-2 v2.1"/>
```

## Fail example
```html
<meta name="content-protocol" content="ICP-Lite v1.0"/>
<meta name="content-protocol" content="ITW-Lite"/>
<meta name="content-protocol" content=""/>
```
Validator emits: `Invalid content-protocol. Expected one of: ICP-Lite v1.4, ICP-2, ICP-2 v2.1. Found "..."`.

## Fix guidance
On new pages, use `ICP-2 v2.1` (the current canonical). On legacy pages mid-migration, `ICP-Lite v1.4` is still accepted. Do NOT leave the tag empty — pick one.

## Migration note
When ICP-2 migration completes, the validator's `ACCEPTED_PROTOCOLS` list shrinks. Update this rule's `check:` field at that point.
