## Rule
`ai-summary` must be ≤ 250 characters total. Within that, the first ~155 characters must form a complete, standalone sentence so an AI assistant truncating at ~155 still gets a meaningful summary.

## Why (rationale)
Dual-cap serves two audiences: AI systems that truncate at their internal limit (~155 chars for many snippet-level uses) and those that take the full summary. Without the first-155 standalone constraint, mid-sentence truncation produces nonsense.

## Pass example
```
Nassau is a tender-free port five minutes' walk from downtown Bay Street. Cable Beach via jitney ($1.25, 25 min); Atlantis by taxi ($8 each way).
```
First period falls at char 68. Total length: 143. Both caps satisfied.

## Fail example
```
Nassau is a tender-free port five minutes walk from downtown Bay Street and offers easy access to Cable Beach by jitney as well as Atlantis by taxi, with the Queen's Staircase, Government House, and Junkanoo Beach all within walking range of the terminal.
```
Length: 266 (over 250). No standalone sentence before 155 chars. Double-fails.

## Fix guidance
Write the summary as two sentences. The first must stand alone with the most important fact. The second adds detail. Keep total under 250.

## Implications

### If validator wins (current behavior: only length checked)
- Validator enforces only `<= 250 chars` (line 670-685). It does NOT check the 155-char standalone-sentence rule.
- Current pages pass validation while silently violating the standards-documented dual-cap rule.
- Pages written for ICP-Lite v1.4 think they're compliant but fail the full spec.
- Downstream AI systems truncating at ~155 chars see mid-sentence garbage on many pages.

### If standards win (promote to full dual-cap validator check)
- Validator gains a new check: locate first-sentence terminator (`.`, `!`, `?` not inside an abbreviation) and require position ≤ 155 AND preceding content is a complete clause.
- Implementation cost: ~30 lines in validate-port-page-v2.js + ship/venue equivalents.
- Expected fallout: an unknown number of existing pages have a 155-char sentence that is NOT standalone (ends with "and" or similar). Audit required before enforcement.
- Writers lose some flexibility; gain consistent AI-consumption quality.

## Recommendation
**Promote the validator to full dual-cap.** The standards already specify the rule; the validator is simply incomplete. Migration path:
1. Add the check at `warn` severity initially so the fallout is counted without blocking commits.
2. Run an audit pass; fix offending pages as a batch (same family of fix across many pages — well-suited to a bulk pass).
3. Flip severity to `error` once fallout < 5 pages.

Justification: this is the exact failure mode the user is tired of — a rule that lives in docs but not in code. Fix the gap in the validator so docs and code align. Estimated work: 2 hours validator + 1 day content audit. Downside if we don't: every new page ships with the same silent defect.
