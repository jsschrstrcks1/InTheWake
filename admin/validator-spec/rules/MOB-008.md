## Rule
The shared stylesheet (`assets/styles.css`) must include a mobile-hardening section containing the baseline mobile safeguards that individual pages rely on (table overflow-x, hero max-width, touch-target sizing, 15px font floor). The validator emits INFO (not error/warn) because the absence is a Phase-2-pending state, not a per-page regression.

## Why (rationale)
The per-page rules (MOB-002 through MOB-007) assume the shared stylesheet provides baseline mobile-safe CSS. If the baseline is absent, pages that look fine in the validator's static scan can still fail in real phones. MOB-008 tracks whether the baseline exists as a site-wide project status signal rather than a per-page requirement.

Info severity intentional — this is a one-time project setup check, not a recurring per-page rule.

## Pass example
`assets/styles.css` contains:
```css
/* === MOBILE HARDENING (v3.010.400) === */
@media (max-width: 640px) {
  .table-responsive { overflow-x: auto; }
  .port-hero img, .hero img, .ship-hero img { max-width: 100%; height: auto; }
  button, a.btn, [role="button"] { min-height: 44px; min-width: 44px; }
  body, p, li { font-size: max(15px, 1rem); }
  /* ... etc */
}
```

## Fail (info) example
Stylesheet doesn't contain any `/* mobile hardening */` block or equivalent. Validator emits: `Mobile hardening CSS section not yet added to assets/styles.css (Phase 2 pending)`.

## Fix guidance
Add the mobile-hardening block once. When the block exists, flip this rule's status to "satisfied fleet-wide" (no further action needed; individual-page rules MOB-002..007 handle recurring enforcement).

## Backfill note
Standards docs don't currently describe what should be in the mobile-hardening block. Phase 6 regenerated standards should specify the baseline contents so future stylesheet work can verify against a spec.
