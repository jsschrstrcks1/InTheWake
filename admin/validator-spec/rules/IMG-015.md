## Rule
An image file appearing in two or more port directories is flagged. Same-port reuse is fine; cross-port reuse is not — a photo of Nassau appearing in the Falmouth directory means one of the two pages is lying about what it's showing.

The `allowedDuplicates` file names a single "approved owner" for shared images that legitimately belong to multiple ports (e.g., an overhead map of the Caribbean). Only the approved owner passes; all other ports using the same hash fail.

## Why (rationale)
Cross-port contamination is the "copy an image from Nassau to pad the St Kitts gallery" failure mode. It produces pages that look complete while lying — the St Kitts gallery shows a Nassau photo. The MD5 cross-port scan catches this deterministically and cheaply. The `allowedDuplicates` exception list handles legitimate sharing.

Standards docs don't describe this; validator-only policy. Backfill in Phase 6.

## Pass example
`ports/img/nassau/nassau-hero.webp` has MD5 `abc123`. No other port directory contains `abc123`. Passes.

## Pass example (allowed duplicate)
`assets/maps/caribbean-overview.webp` (MD5 `def456`) is listed in allowedDuplicates with approved owner `caribbean-overview`. Referenced by both Nassau and Falmouth pages — passes (owned by the overview page; used by others).

## Fail example
`ports/img/falmouth/falmouth-beach.webp` has MD5 `abc123`. That hash also exists in `ports/img/nassau/` (where it originated). allowedDuplicates doesn't list Falmouth as approved. Validator emits: `Cross-port image duplication: <filename> also appears in nassau`.

## Fix guidance
Source a real Falmouth photograph, attributed and verified. Or, if the image is legitimately shared (both ports genuinely use the same map), add an entry to `admin/allowed-duplicates.json` naming a canonical owner. Do NOT resolve by deleting the `allowedDuplicates` entry — that just masks the violation.
