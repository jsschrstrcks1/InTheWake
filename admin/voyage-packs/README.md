# Voyage Packs — Workflow & Lifecycle

**Purpose:** This directory holds the source files and deliverables for In the Wake's Voyage Packs (W12 of the v2.5 plan). This README documents the full lifecycle so the next person — Ken, a future contributor, or me on a future session — can pick it up cold.

**Last updated:** 2026-05-09

## What's in this directory

For each pack, three files in lockstep:

| File | Role | Audience | How it's produced |
|---|---|---|---|
| `vN.N-<pack-slug>.md` | Source of truth — every fact, every paragraph | Author (Ken) | Hand-written / edited |
| `vN.N-<pack-slug>.html` | Offline-PWA-capable rendered version with the fillable Emergency Contacts handoff card, print + PDF buttons, floating TOC sidebar | Buyer (browser, after first load works offline) | Hand-translated from the `.md` matching the v0.1.2 NCL Aqua HTML conventions |
| `vN.N-<pack-slug>.pdf` | Print-friendly paginated deliverable | Buyer (download from payment processor) | Generated by `admin/scripts/voyage-pack-pdf-build.sh` from the `.md` |

The `.md` is the source of truth. If the three diverge, **the `.md` wins** and the others must be regenerated.

## The lifecycle

```
┌─────────┐      edit       ┌─────────┐    regenerate   ┌─────────┐
│   .md   │ ──────────────► │  .html  │                 │   .pdf  │
│ (source │                 │(offline │                 │(deliver-│
│  truth) │ ──────────────► │  PWA)   │                 │  able)  │
└─────────┘                 └─────────┘                 └─────────┘
     │                                                        │
     │     pre-commit hook enforces .pdf stays in sync        │
     └────────────────────────────────────────────────────────┘
```

### Day-to-day flow

1. **Edit the `.md`.** Add a fact, fix a number, restructure a section, etc.
2. **Edit the `.html`** to match. The translation is hand-done, not automated, because the HTML version has structural elements the markdown doesn't (TOC sidebar, swiper, fillable form fields). When the `.md` adds or removes a section, the `.html` needs the matching change.
3. **Regenerate the `.pdf`** by running `admin/scripts/voyage-pack-pdf-build.sh`. The script is idempotent — running it on an unchanged tree does nothing.
4. **Commit all three files together.** The pre-commit hook blocks committing a modified `.md` without a regenerated `.pdf`.
5. **Upload the new `.pdf` to the payment processor's product entry** (Gumroad / Lemon Squeezy / Stripe). This is the only manual step that connects the repo to buyers.

### What the pre-commit hook enforces

`.githooks/pre-commit` check #4 runs `voyage-pack-pdf-build.sh --check` whenever a voyage-pack `.md` file is staged. If any pack's `.pdf` is missing or older than its `.md`, the commit is blocked with this message:

```
pre-commit: stale or missing Voyage Pack PDF(s). Commit blocked.
  Rebuild:   admin/scripts/voyage-pack-pdf-build.sh
  Bypass:    git commit --no-verify  (only when intentionally shipping a .md change without a regenerated .pdf)
```

The hook intentionally does NOT enforce the `.html` ↔ `.md` sync — that's a content-judgment task, not a mechanical one, and an over-zealous check would block legitimate work. The author is trusted to keep them in sync.

## The build script

### What it does

`admin/scripts/voyage-pack-pdf-build.sh`:

1. Detects an installed pandoc PDF engine (preference order: weasyprint, wkhtmltopdf, xelatex, pdflatex).
2. For each pack source `.md`:
   - If the corresponding `.pdf` exists and is newer than the `.md` → skip (idempotent default).
   - Otherwise, run pandoc to generate the `.pdf` using the `voyage-pack-print.css` stylesheet.
3. Reports status (built / skipped / failed) per pack.

### Usage

```bash
# Build any stale packs (default)
admin/scripts/voyage-pack-pdf-build.sh

# Force rebuild even if PDF is newer
admin/scripts/voyage-pack-pdf-build.sh --force

# Check for stale PDFs without building (used by pre-commit)
admin/scripts/voyage-pack-pdf-build.sh --check

# Build only one pack
admin/scripts/voyage-pack-pdf-build.sh symphony
admin/scripts/voyage-pack-pdf-build.sh ncl-aqua

# Combine flags
admin/scripts/voyage-pack-pdf-build.sh --force symphony

# Print the help block
admin/scripts/voyage-pack-pdf-build.sh --help
```

### Exit codes

- `0` — success (or `--check` passes — no stale PDFs)
- `1` — at least one pack failed to build, or `--check` found a stale PDF, or pandoc/engine not installed
- `2` — bad command-line argument

## Engine setup (one-time)

### macOS

```bash
brew install pandoc
pip3 install weasyprint
```

### Debian / Ubuntu

```bash
sudo apt install pandoc
pip3 install weasyprint
# OR if pip3 weasyprint won't install (system Python is too restrictive):
sudo apt install wkhtmltopdf
```

### Verify

```bash
admin/scripts/voyage-pack-pdf-build.sh --check
```

If it reports "PDF MISSING" for both packs, you're ready to build.

If it reports "No PDF engine found," install one of the engines above and retry.

## The print stylesheet

`voyage-pack-print.css` controls the PDF's typography and pagination. It uses the brand palette from the main site:

- Headings: Helvetica Neue / Arial sans-serif, `--sea` color (`#0a3d62`)
- Body: Georgia serif, `--ink` color (`#083041`)
- Page size: US Letter, 0.85" margins
- Page-break-before on every `<h2>` (each section starts a new page)
- Footer: page number + site URL (suppressed on the first page)
- Tables: bordered, `--foam` background on header rows
- Code blocks: monospace, light beige background

Edit the CSS file and re-run `--force` to regenerate the PDFs with new styling.

## Adding a new pack

When a third pack is created (custom pack, additional generic itinerary, etc.):

1. **Author the `.md`** in this directory using the existing two as templates. Match the section structure: §1 Ship, §2 Day-by-Day, §3 Pre-Cruise Countdown, §4 Packing, §5 Budget, §6 Emergency Contacts, §7+ as needed.
2. **Translate to `.html`** matching the `v0.1.2-ncl-aqua-veterans-solo-group-dec-2027.html` conventions:
   - Same `<head>` template (ICP-Lite metas, OG/Twitter cards, JSON-LD)
   - Same TOC sidebar pattern with `vp-toc` class
   - Same fillable Emergency Contacts handoff card with a unique `data-storage-key`
   - Same print + PDF buttons with the same scope attributes
   - Wire `/assets/css/handoff-card.css`, `/assets/css/handoff-card-print.css`, `/assets/js/handoff-card.js`, `/assets/js/pdf-download.js`
   - Set `<meta name="vp-filename-base" content="inthewake-<short-name>"/>` for the PDF filename
3. **Add it to the build script.** Open `admin/scripts/voyage-pack-pdf-build.sh` and:
   - Add a constant near the top: `NEW_PACK_MD="$PACKS_DIR/<your-slug>.md"`
   - Add a target case in the `case "$target"` block at the bottom with a `build_pack` call
   - Add an entry in `run_check_only`'s loop
4. **Update `/voyage-packs.html`** to add the new pack as an `<article class="vp-product-card">` with the buy button.
5. **Document the new pack** in this README's "Current packs" table.
6. **Build the PDF** with `admin/scripts/voyage-pack-pdf-build.sh <new-slug>`.
7. **Upload the PDF and HTML zip** to the payment processor as a new product.
8. **Update `admin/W12-PRODUCT-LAUNCH-CHECKLIST.md`** if the new pack changes any operational item.

## Common errors and fixes

### "pre-commit: stale or missing Voyage Pack PDF(s). Commit blocked."

You changed a pack `.md` and forgot to regenerate the `.pdf`.

```bash
admin/scripts/voyage-pack-pdf-build.sh
git add admin/voyage-packs/<your-pack>.pdf
git commit  # retry; should pass now
```

If the `.md` change was intentional but you don't want to ship the PDF in this commit (e.g., you're staging the change for a later batch), use `git commit --no-verify` to bypass. Be honest with yourself: that's a real exception, not a default move.

### "✗ pandoc is required. Install: …"

You haven't installed pandoc. See the engine setup section above.

### "✗ No PDF engine found. Install ONE of: …"

Pandoc is installed but no PDF engine is. Install weasyprint, wkhtmltopdf, or texlive (in that preference order) per the engine setup.

### PDF builds but the typography looks wrong

Likely engine difference. weasyprint and wkhtmltopdf use the CSS file; xelatex and pdflatex use template variables and ignore the CSS. If your output came from a LaTeX engine but you want CSS-styled output, install weasyprint.

### "PDF STALE" but the `.md` hasn't changed

File-system mtime jitter (e.g., from a `git checkout` that touched the `.md` modification time). Run `admin/scripts/voyage-pack-pdf-build.sh --force` to force-rebuild and clear the staleness flag.

### Pre-commit hook isn't running at all

Confirm the hooks path is configured:

```bash
git config --get core.hooksPath
# Should print: .githooks
```

If empty, set it:

```bash
git config core.hooksPath .githooks
```

## What this workflow does NOT do

- **Auto-publish.** The PDF doesn't auto-upload to the payment processor. That's a manual step (deliberately — re-uploads matter for buyers and shouldn't happen by accident).
- **Auto-translate `.md` to `.html`.** The HTML translation is hand-done because the HTML version has structural elements the markdown can't represent. A future automation pass could template parts of it (e.g., metadata block, TOC), but the editorial translation stays manual.
- **Bundle the zip.** The W12 launch checklist's bundling step (PDF + HTML into one zip per pack with a versioned filename) is still a manual step. Could be automated later if needed.
- **Validate the `.html` ↔ `.md` sync.** The two are kept in sync by the author, not enforced by the hook. A future check could compare section headings, but it's not worth it at the current pack count.
