## 1. What Phase 1.5 covers (and what it doesn't)

Phase 1.5 is the **fleet-wide image authenticity audit**. It addresses three failure classes the original plan named together as "the 891 placeholder file problem," plus a fourth that this session surfaced:

| Failure class | Detector | What it catches |
|---|---|---|
| **Flickr-889 placeholder stubs** | `sourcing.py audit-attr <slug>` | attr.json with `license: "Flickr (verify license)"`, empty `source_url`, `source_type: "Flickr public feed"` |
| **Source-URL low diversity** | `sourcing.py audit-attr <slug>` + validator ATTR-003 | 4+ images sharing 1-2 source URLs (the original Flickr-889 escape pattern) |
| **Wrong-subject Wikipedia conflation** | per-file `Read` | filename says the right port but the image shows a different place (e.g., Scottish Shetland filed as South Shetland Islands) |
| **Slot-vs-subject mismatch** | per-file `Read` + HTML alt cross-check | image is correct geography but filename doesn't match content (e.g., NASA satellite filed as "food.webp") |

The first two are detected mechanically. The third and fourth require a human eye on the bytes.

**Out of scope for Phase 1.5:**
- Sourcing new images for ports already meeting the 11-image floor — that's a richness improvement, not an authenticity repair.
- Refactoring port HTML structure or content sections — those are content tasks.
- Anything that touches ship pages or venue pages — Phase 1.5 is scoped to port images only.

## 3. Phase 1.5 sub-phases

Five sub-phases. Each commits independently. Each has its own exit criteria.

### Phase 1.5a — Finish the 53 validator `missing_image_file` ports (highest urgency)

**Goal:** Drive the validator's `missing_image_file` blocker count from 49 → 0 (4 already done this session).

**Method:** Continue the one-port-at-a-time pattern proven on callao + ushuaia + zadar + south-shetland-islands. For each:
1. `sourcing.py inventory <slug>` + `sourcing.py audit-attr <slug>` + `sourcing.py reuse-check <slug>`
2. For every wrong-subject hint (placeholder attr, low diversity, suspect filename): `sourcing.py audit-port-images <slug>` to force per-file Read
3. Source or rewrite each broken HTML ref per the proven workflow
4. `sourcing.py validate <slug>` to confirm image-section blockers cleared

**Order:**
- 8 ports with 1 broken ref (5 still pending: port-everglades, port-miami, royal-beach-club-antigua, santa-marta, sydney-ns)
- 15 ports with 2-3 broken refs (cochin, gibraltar, guam, harvest-caye, honningsvag, ilhabela, kiel, messina, montreal, okinawa, port-arthur, puerto-montt, tobago, torshavn, trinidad)
- 14 ports with 4-7 broken refs (colombo, doubtful-sound, gatun-lake, genoa, glacier-alley, guadeloupe, hilo, holyhead, isafjordur, jacksonville, kagoshima, salvador, san-diego, trieste)
- 14 ports with 8-12 broken refs (catania, dubrovnik, hakodate, jakarta, manila, muscat, mykonos, ocho-rios, punta-del-este, reykjavik, san-francisco, santorini, sydney — note: callao already done)
- 2 ports with 13+ broken refs (hong-kong: 18, singapore: 16)

**Sustained pace observed in this session:** ~3-5 images per careful session-hour (including search-license-fetch-Read-convert-attr-validate-commit). Estimated 1.5a completion time: **~3-5 working weeks of careful sourcing**.

**Exit criteria:**
- Validator `missing_image_file` blocker count: 0 fleet-wide
- Every replacement image registered in `admin/sourced-images-registry.json`
- Per-port audit md exists for any port where wrong-subject files were found

### Phase 1.5b — Targeted Read-pass on `audit-attr` placeholder ports (302 ports, 891 files)

**Goal:** Verify or replace every placeholder-shape attr.json. Surface the wrong-subject failure rate beyond `audit-attr`'s reach.

**Method:**
1. Sort the 302 ports by `audit-attr` severity: ports failing both placeholder-shape AND ATTR-003 source-diversity first (180 ports), then placeholder-shape only (122 ports).
2. For each port:
   - `sourcing.py audit-port-images <slug>` to scaffold the audit md
   - Read every file in the directory (typically 8-12 files per port)
   - Classify per file: `correct` / `wrong-subject` / `placeholder` / `unclear`
   - Wrong-subject + placeholder verdicts → delete and either re-source or drop the HTML ref
   - For `unclear`: keep with `subject_verified: unclear` in attr.json + flag in `admin/UNFINISHED_TASKS.md`
   - For `correct`: `verify-flickr` the original Flickr URL (if any survives the stub) and rewrite the attr.json with canonical fields if the license is genuinely confirmed; if not, treat as `unclear`

**Throughput estimate:** Read-pass over 8-12 files per port × 302 ports = **~2,700-3,600 individual Read operations**, each requiring 30-60 seconds of attention. At 60 file Reads per careful hour: **45-60 hours of focused work**.

**Pacing recommendation:** Cap at **20 ports per day**. Past 20 ports the eye glazes and the wrong-subject signal gets weaker. Plan for ~15 working days.

**Exit criteria:**
- All 891 placeholder attr.json files either:
  - (a) Replaced with canonical attr.json from verified source, OR
  - (b) Marked `subject_verified: unclear` and flagged for later, OR
  - (c) Deleted (file + attr.json + HTML refs cleaned up)
- Phase-wide audit report at `admin/audit-reports/phase-1-5b-summary-<date>.md` totaling per-port verdicts

### Phase 1.5c — Validator hardening: catch wrong-subject more cheaply

**Goal:** Move the wrong-subject and slot-vs-subject failure detection out of human eyes and into automation where possible.

**Two new validator rules to propose** (additions to `admin/validator-spec/rules/`):

- **IMG-016 (new): Placeholder license string detector.** Flag any port attr.json with `license` containing `"verify license"` or `source_type` equal to `"Flickr public feed"`. The harness's `audit-attr` already does this; promoting it to the canonical validator means CI catches it on every commit. Severity: `error`.

- **IMG-017 (new): Filename-content slot consistency.** For known-meaning slot suffixes (`-food.webp`, `-harbor.webp`, `-landmark.webp`, `-hero.webp`), require the HTML alt text on the corresponding `<img>` to contain at least one keyword consistent with the slot name (food/cuisine/dish for `-food`, harbor/port/dock for `-harbor`, etc.). Severity: `warn` (since rename pivots like south-shetland-islands/food.webp → iceberg-a23a.webp would otherwise become busywork). This is a hint, not a hard gate.

**Limits of automation:** Neither rule can catch Wikipedia conflation by string-level means (Scottish-Shetland-filed-as-South-Shetland looks correct in text). The only real defense is per-file Read by a human-in-the-loop. Phase 1.5c surfaces what *can* be caught cheaply; the rest stays in 1.5b.

**Implementation:** Each new rule gets a `admin/validator-spec/rules/IMG-<n>.md` file plus an implementation in `admin/validate-port-page-v2.js`. End-to-end run against the fleet baseline before the rules go live, so we have a count of how many ports they would fire on.

**Exit criteria:**
- IMG-016 + IMG-017 specs written, validator code implemented, baseline-impact count recorded
- Both rules wired into `.githooks/pre-commit` and the post-write hook

### Phase 1.5d — Cross-port image-reuse sweep

**Goal:** Find byte-level cross-port duplicates outside `admin/cross-port-image-allowlist.json` and either allowlist them or replace one of the duplicates.

**Method:**
1. Run a fleet-wide md5 scan across all 397 port image directories (extend the harness's `reuse-check` to a fleet mode, or write a one-off script using `admin/check-image-reuse.cjs`).
2. For each duplicate set:
   - Read each duplicate file with the Read tool
   - Decide which port the image truly belongs to (visual identifiers, accompanying caption)
   - If multiple ports legitimately share the image (e.g., a Greenwich Island shot used in both south-shetland-islands and antarctic-peninsula pages): add to the allowlist with reviewer name + date
   - Otherwise: replace one of the duplicates

**Likely findings (extrapolating from the 2026-05-10 image-reuse-report):** the placeholder-ship.webp clone across christchurch, durban, luanda, mombasa, port-moresby is already known. There may be a similar shape across other Antarctic/sub-Antarctic ports given the wrong-archipelago pattern surfaced here.

**Exit criteria:**
- 0 cross-port duplicates outside the allowlist
- Allowlist reviewed and updated with this session's findings

### Phase 1.5e — Fleet-wide verification sweep

**Goal:** Final pass after 1.5a-d to confirm the fleet is image-clean.

**Method:**
1. `node scripts/batch-validate.js --json` — confirm 0 `missing_image_file` blockers and 0 ATTR-003 failures
2. `node admin/check-image-reuse.cjs` — confirm 0 unauthorized cross-port duplicates
3. `sourcing.py audit-attr` on every port (write a fleet-mode invocation if needed) — confirm 0 placeholder shapes remain
4. Spot-Read 5% of newly-canonical attr.json files at random; confirm content + license claims still match
5. Cross-check `admin/sourced-images-registry.json` count against the number of replacement webps committed in this branch's commits

**Exit criteria:**
- All four sweeps clean
- Fleet-wide summary report written: `admin/audit-reports/phase-1-5-complete-<date>.md`
- Update `admin/IMAGE_SOURCING_WORKFLOW.md` with any rules learned during 1.5a-d that should become standing process

## 4. Sequence and dependencies

```
Phase 1.5a (53 broken-ref ports)
        ↓ informs likely failure patterns
Phase 1.5b (302 placeholder-shape ports, per-file Read)
        ↓ surfaces wrong-subject rate, informs detector design
Phase 1.5c (IMG-016 + IMG-017 validator rules)
        ↓ guards against regression
Phase 1.5d (cross-port reuse sweep)
        ↓ deduplicates after 1.5a/b replacements
Phase 1.5e (verification sweep) — confirms clean state
```

Phases 1.5a and 1.5b can run in parallel if work is split between two careful sessions: 1.5a deals with the validator-flagged broken-ref ports (urgency), 1.5b deals with the latent-placeholder ports (richness/integrity). Phases 1.5c-e are sequential.

## 5. Risks and how to manage them

| Risk | Likelihood | Mitigation |
|---|---|---|
| **Pace drift to clever-not-careful** under volume (the original drift this whole replan exists to address) | High at scale | Hard 20-port-per-day cap on Phase 1.5b. Spot-check 10% of finished ports the day after, with fresh eyes. Mandatory `sourcing.py audit-port-images` before declaring any port done. |
| **CC source pool drying up** as Phase 1.5a touches less-photographed ports | Medium | Pre-flight: for each Phase 1.5a port, do the search-and-verify step up-front before starting the replacement. If 0 CC candidates exist for the slot, document and flag rather than push through. |
| **LoC tile.loc.gov flakiness** (already observed: HTTP/2 truncation, occasional 503s on larger sizes) | Medium | Harness already forces `--http1.1`; the retry pattern with smaller variant sizes is proven. No additional mitigation needed. |
| **Wikimedia Commons egress blocked** | Certain (already observed) | Phase 1.5b will surface many existing Commons-sourced files — verifying their license requires the source URL but Commons is unreachable. Mitigation: trust the existing source URL string when it points to a specific Commons File: page (verifiable in production but not in this sandbox), but downgrade subject verification to per-file Read since we cannot re-confirm against Commons. |
| **Subagent verification temptation** under volume | Medium | The 2026-05-10/11 sub-agent ship-image cleanup is precedent for what this would look like. Per CAREFUL.md "Subagent Output is a Hypothesis": if subagents are used to parallelize 1.5b, mandatory 10% spot-Read by human + dry-run on one port + 5% diff after cleanup. The hardening plan's Rule A (audit every file when one is wrong) does not transfer cleanly to sub-agents — they don't have the same model of the directory state. Prefer doing 1.5b in-session even at the cost of throughput. |
| **Registry inflation** | Low | The registry grows with every committed image — could reach 4,000-5,000 entries by Phase 1.5e. JSON file remains tractable; no DB migration needed. |
| **"Trust" rebuilds quietly across sessions** | Medium | The photographer log (`admin/sourced-photographers-log.json`) is the durable record. Every new session reads it at start and continues the spot-check cadence. |

## 6. What this plan deliberately leaves out

- **Ship-page images.** The 2026-05-10/11 audit already cleaned the most egregious flickr-named ship images. Anything remaining there is a separate workstream.
- **Venue / restaurant images.** Phase 1.5 is port-scoped. Venue images get their own audit.
- **Cross-language captioning.** The site is English-only; alt text in Spanish or Croatian in the existing attr files (e.g., Carlos Esquives's "Noche en el Parque del Amor") is preserved in the source attribution but the production alt text stays English.
- **Performance optimization.** Phase 1.5 may surface that some images are oversized (e.g., 264 KB webp where 50 KB would do); deferred. The bytes-on-disk problem is correctness first, file size second.

## 7. Decision points the user owns

Before committing to start Phase 1.5b at the 302-port scale, confirm:

1. **In-session vs subagent for 1.5b.** Throughput trade-off described in the Risks table. My default per the hardening plan is in-session.
2. **20 ports/day cap acceptable?** Or smaller? Or larger if Phase 1.5a goes faster than estimated?
3. **IMG-016 + IMG-017 severity.** I proposed `error` for IMG-016 (placeholder licenses) and `warn` for IMG-017 (filename-slot). Confirm or downgrade.
4. **`subject_verified: unclear` in production attr.json — acceptable, or stop-the-line?** My read: acceptable for honest documentation, escalates pressure to find a verified replacement but does not block production. The alternative — deleting any unclear-subject file — would shrink galleries by a lot and create more empty-carousel pages than the 2026-05-10/11 incident did.
