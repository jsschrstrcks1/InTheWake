## Rule
Ship pages should contain at least one ARIA live region — an element with `role="status"`, `role="alert"`, or an `aria-live` attribute. Live regions let screen readers announce dynamic content changes (form errors, toast messages, carousel slide changes) without requiring user focus to move.

## Why (rationale)
Pages with interactive components (carousels, live-tracker iframes, date pickers, form validation) change content without navigation. Without a live region, screen reader users miss those updates. One live region per page covers the majority of announcements.

Ship validator enforces at warn; port and venue don't. Standards silent.

## Pass example
```html
<!-- Anywhere in the page, typically near the top: -->
<div id="a11y-status" role="status" aria-live="polite" class="visually-hidden"></div>

<!-- Then JS writes status messages to it: -->
<script>
  document.querySelector('#a11y-status').textContent = 'Loaded 10 photos';
</script>
```

## Fail (warning) example
No element with role="status" / role="alert" / aria-live anywhere. Validator emits: `No ARIA live region found`.

## Fix guidance
Add a single visually-hidden live region to the page template. `role="status"` + `aria-live="polite"` is the default-safe pair. Use it for non-critical updates; reserve `role="alert"` or `aria-live="assertive"` for actual errors (which rarely exist on static content pages).
