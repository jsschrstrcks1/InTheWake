## Rule
Production JavaScript source (any file served to end-users) must not contain `console.log(...)` calls. Admin tooling, development scripts, test files, and validator code are exempt. The check is a direct pattern match on `console\.log\(`.

## Why (rationale)
Console noise in production is a low-grade leak — it surfaces internal variable names, data shapes, and debug assumptions to anyone who opens DevTools. Competitors scraping for intel see the data shape; privacy-concerned users see what's being tracked; and it's a reliable tell that code wasn't cleaned before shipping.

Warn severity (not error) reflects the reality that legacy code sometimes carries these; the warn makes cleanup visible without blocking every commit. The project-level `NEVER console.log/warn in production` rule in CLAUDE.md is the hard intent; the validator is a helper not the enforcer.

## Pass example
```js
// production file — no console.log
function initCalculator() {
  document.querySelector('#total').textContent = formatUSD(0);
}
```

```js
// admin/script.cjs — exempt
console.log(`Processed ${files.length} files`);
```

## Fail (warning) example
```js
// assets/js/drink-calculator.js — production file
function onSubmit() {
  console.log('User clicked submit', formData); // WARNING
  ...
}
```
Validator emits: `Avoid console.log in production code`.

## Fix guidance
Delete the call. If the log captured useful state, consider adding a feature-flagged diagnostic that only runs when an explicit debug switch is set. Do NOT leave the log "for now" — the AUDIT_REPORT_2025_11_19 found 12 such instances that had accumulated across production files.
