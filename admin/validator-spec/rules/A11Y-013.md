## Rule
For each structural element (`<div>`, `<section>`, `<article>`, `<header>`, `<footer>`, `<nav>`), the count of opening tags must equal the count of closing tags in the document source. Mismatched counts indicate missing or extra closing tags, which produce malformed DOM trees at parse time.

## Why (rationale)
Browsers attempt to recover from unclosed tags (pushing implicit close tags at block boundaries), but the recovery is browser-dependent — Firefox, Safari, Chrome, and screen readers can produce different DOM trees from the same invalid HTML. Deterministic accessibility requires a valid DOM; tag balance is the coarsest validity check.

## Pass example
Section count balanced:
```html
<section>
  <h2>Dining</h2>
  <p>...</p>
</section>  <!-- 1 open, 1 close -->
<section>
  <h2>Shore excursions</h2>
</section>  <!-- 2 open, 2 close -->
```

## Fail example
```html
<section>
  <h2>Dining</h2>
  <section>  <!-- nested section opens -->
    <h3>Specialty restaurants</h3>
  <!-- forgot to close inner section -->
</section>  <!-- only closes outer, inner was never closed -->
```
Validator emits: `Mismatched section tags: 2 opening, 1 closing` (or similar).

## Fix guidance
Run the file through `tidy -q -e <file>` or equivalent HTML validator. Common causes: copy-paste errors across batch edits, conditional rendering paths that emit one branch's opening tag and another's closing tag.
