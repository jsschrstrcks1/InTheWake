## Rule
If `Review.reviewRating.ratingValue` is declared, the validator emits a warning — **regardless of the value**. The blanket-warn stance exists because audits have found ratings that were never actually assigned editorially; they were generated by templates or inherited from prior drafts. Until every ship page has a documented editorial rating basis, every rating is flagged.

## Why (rationale)
A declared 4.5-star rating in structured data is picked up by rich-result systems and AI assistants. If the site has never actually done a structured ratings exercise, those numbers are fabrications — worse than omitting the rating entirely, because they signal false confidence. The blanket warn is a holding pattern: it nudges the team to either (a) do the rating work seriously, or (b) remove ratingValue until they do.

Standards docs don't describe this stance — it emerged from audits. Backfill in Phase 6 should capture the policy.

## Pass example
```html
<script type="application/ld+json">
{ "@type": "Review",
  "itemReviewed": { "@type": "Product", "name": "Allure of the Seas" },
  "author": { "@type": "Person", "name": "In the Wake Editorial" },
  "reviewBody": "..."
}
</script>
```
No ratingValue present. No warning.

## Fail (warning) example
```html
{ "@type": "Review",
  "reviewRating": { "@type": "Rating", "ratingValue": 4.5, "bestRating": 5 }
}
```
Validator emits: `Review has ratingValue 4.5 — must be based on real editorial assessment, not templated`.

## Fix guidance
Preferred: remove `reviewRating` from the JSON-LD until a documented editorial rating rubric exists. If ratings are going to stay, establish a rubric document (criteria, scale, who sets the rating, when it's reviewed) and reference it from the rule's `standards-source:` once written. The warning stays until the policy exists; removing the warning without doing the work is the "clever, not careful" move this spec explicitly rejects.
