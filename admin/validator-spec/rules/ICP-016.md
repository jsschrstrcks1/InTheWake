## Rule
The canonical URL (see ICP-008 for presence) must be absolute and point to the `https://cruisinginthewake.com/` origin. Relative canonical URLs, http://, or non-cruisinginthewake.com hosts all fail.

## Why (rationale)
The site is served from `cruisinginthewake.com` (CNAME confirms). Relative canonicals resolve differently in some crawl contexts. A non-cruisinginthewake.com canonical tells search engines the page's authoritative copy is elsewhere — which is wrong and actively harmful for indexing.

## Pass example
```html
<link rel="canonical" href="https://cruisinginthewake.com/ports/nassau.html"/>
```

## Fail example
```html
<link rel="canonical" href="/ports/nassau.html"/>
<link rel="canonical" href="http://cruisinginthewake.com/ports/nassau.html"/>
<link rel="canonical" href="https://example.com/ports/nassau.html"/>
```
Venue validator (T18): `Canonical URL must be absolute https://cruisinginthewake.com/ format, found "..."`.

## Fix guidance
Always write the full `https://cruisinginthewake.com/...` URL. A pre-commit check or build step could mechanize this; currently it's manual.

## Coverage note
Only the venue validator (T18) enforces the absolute format today. Port and ship validators check presence (ICP-008) but not the domain. Phase 2 extraction should propagate T18's check to port + ship. Treat that as a post-Phase-2 follow-up issue rather than a change to the spec.
