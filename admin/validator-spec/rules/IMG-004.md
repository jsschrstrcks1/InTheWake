## Rule
Every image below the fold (anything that isn't the hero) must declare `loading="lazy"`. Missing `loading="lazy"` on a non-hero image forces the browser to download it during initial page load, hurting LCP and consuming mobile bandwidth unnecessarily.

However, the port and ship validators disagree on severity.

## Why (rationale)
Core Web Vitals. A port page has 11-25 images (IMG-005, IMG-006). If none are lazy-loaded, the browser fetches all of them on first paint — catastrophic on mobile. Lazy loading defers everything off-screen until it's near the viewport.

## Pass example
```html
<!-- Hero: eager, high priority -->
<img class="port-hero-img" src="..." loading="eager" fetchpriority="high" alt="..."/>

<!-- Gallery and body images: lazy -->
<img src="..." loading="lazy" alt="..."/>
<img src="..." loading="lazy" alt="..."/>
```

## Fail example
```html
<img class="port-hero-img" src="..." fetchpriority="high" alt="..."/>
<img src="..." alt="..."/>  <!-- no loading attribute — non-hero, fails -->
<img src="..." alt="..."/>
```
Port validator: `N non-hero images missing loading="lazy"` (BLOCKING).
Ship validator: `N images missing loading="lazy"` (WARNING).

## Implications

### If port's BLOCKING severity wins (propagate to ship + venue)
- Ship validator updated at line 1514 to push to `errors` instead of `warnings`.
- Consistent severity across page types. Consistent message to writers.
- **Risk:** any ship page that currently passes via warning now fails. Audit needed — how many ship pages have non-lazy non-hero images today?
- Migration: bulk grep-and-fix. Moderate cost.

### If ship's WARNING severity wins (relax port to match)
- Port validator lines 1203-1223 demoted to `warnings`.
- Lower enforcement; more pages ship with LCP regressions.
- Performance audit becomes harder — site-wide Lighthouse scores drift as non-hero images lack lazy loading.
- Cheaper short-term, silently worse long-term.

## Recommendation
**Port severity wins — promote ship's check to BLOCKING.**

Reasoning:
1. LCP and total-page-weight are real performance metrics that real mobile users experience. Warnings that never turn into errors tend to accumulate silently.
2. The fix is mechanical (add `loading="lazy"` to every non-hero `<img>`); the audit cost is bounded.
3. Consistency across validators is a separate virtue — three page types, three different severity levels on the same rule is exactly the sprawl the spec exists to catch.
4. Counter-argument: if there's a real reason some ship pages legitimately should NOT lazy-load (above-fold carousel, for instance), the right fix is to explicitly mark those elements as hero-class, not to soften the rule globally.
