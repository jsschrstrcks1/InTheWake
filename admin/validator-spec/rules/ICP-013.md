## Rule
A page's WebPage JSON-LD `dateModified` field must exactly match the `last-reviewed` meta content string. ICP-2 v2.1 preserved this as an exact-match requirement even while relaxing the `description` parity rule (see ICP-014).

## Why (rationale)
Dates are machine-comparable; relaxing exact-match adds no value and opens timezone/format drift bugs. An AI consumer reading the page should see the same ISO date in both places — anything else is a sign of stale build output.

## Pass example
```html
<meta name="last-reviewed" content="2026-04-15"/>
...
<script type="application/ld+json">
{ "@context": "https://schema.org", "@type": "WebPage",
  "name": "Nassau, Bahamas — Cruise Port Guide",
  "dateModified": "2026-04-15"
}
</script>
```

## Fail example
```html
<meta name="last-reviewed" content="2026-04-15"/>
...
<script type="application/ld+json">
{ "@type": "WebPage", "dateModified": "2026-04-10" }
</script>
```
Validator emits: `Schema dateModified (2026-04-10) must match last-reviewed (2026-04-15)`.

## Fix guidance
When you bump `last-reviewed`, bump `dateModified` in every JSON-LD block on the same page. A mechanical update — ideal for a build script. Until that exists, enforce via pre-commit habit.
