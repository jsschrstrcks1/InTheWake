## Rule
The validator currently treats `ai-breadcrumbs` HTML comment blocks as an anti-pattern and warns on their presence. The standards docs describe `ai-breadcrumbs` as a required feature for entity pages (ships, ports, restaurants). These positions are incompatible. User must decide.

## Why (rationale)
`ai-breadcrumbs` were introduced in ICP-Lite v1.0/v1.4 as HTML comment blocks inside entity pages, providing structured hints to AI consumers (entity type, parent URL, siblings, intended reader). The spec is still in `.claude/ONBOARDING.md` and `new-standards/v3.010/AI_BREADCRUMBS_SPECIFICATION.md`. Somewhere between ICP-Lite v1.4 and ICP-2 v2.1, someone decided HTML comments are unreadable to crawlers and the metadata should live in JSON-LD instead. The port validator was updated to warn on their presence; the standards docs were not updated to match.

This is exactly the "validators grew, standards didn't" drift the spec exists to surface.

## Pass example (by validator)
```html
<!-- no ai-breadcrumbs comment block -->
<script type="application/ld+json">
{ "@type": "Place", "name": "Port of Nassau", "containedInPlace": { "@type": "Country", "name": "Bahamas" } }
</script>
```

## Pass example (by standards docs)
```html
<!-- ai-breadcrumbs
     entity: Port of Nassau
     type: Port Information Page
     parent: /ports.html
     category: Caribbean Ports
     cruise-lines: Royal Caribbean, Carnival, Norwegian, Celebrity, MSC, Disney
     updated: 2026-04-15
     -->
```

## Fail example (under validator)
Any page containing the string `ai-breadcrumbs` — validator emits: `ai-breadcrumbs HTML comments are not read by any crawler — remove them`.

## Fix guidance (per current conflict)
Until the conflict is resolved, either choice leaves a warning somewhere — either the validator warns or the standards docs describe a feature you're not using. Do not silently pick; wait for the user's decision.

## Implications

### If validator wins (remove all ai-breadcrumbs; update standards to match)
- **Current pages affected:** entity pages across ships/*, ports/*, restaurants/* that include the block. Likely hundreds. A repo-wide grep would give an exact count before any change.
- **Removal cost:** A straight regex strip of the `<!-- ai-breadcrumbs...-->` block from every entity page. Moderate, not hard.
- **Standards docs affected:** `.claude/ONBOARDING.md` AI-Breadcrumbs section (lines 280-295), `new-standards/v3.010/AI_BREADCRUMBS_SPECIFICATION.md` (whole file — would be deprecated/archived), `.claude/skills/standards/resources/ai-breadcrumbs-spec.md`, references in `admin/claude/CLAUDE.md`.
- **Downstream:** any generator that emits ai-breadcrumbs must be updated. Any SKILL.md that references them must be updated.
- **Value lost:** arguably nothing — HTML comments genuinely aren't parsed by crawlers; and LLM consumers don't need the comments either because `mainEntity` JSON-LD declares the same structure in a read format.
- **Risk:** none to readers. Mild risk to any offline LLM tooling built to parse the comments directly.

### If standards win (keep ai-breadcrumbs; remove the validator anti-pattern check)
- **Validator change:** delete lines 753-760 of validate-port-page-v2.js (and equivalent in ship/venue). Trivial.
- **Content cost:** zero — entity pages keep their current blocks.
- **Value preserved:** ai-breadcrumbs are a human-readable inline hint for developers + Claude sessions looking at the HTML. Claude reading a ship page's raw HTML does see the comment and does use it as context. That has measurable value even if no web crawler reads it.
- **Risk:** the rationale originally given for removal — "not read by any crawler" — is factually true. The rule's force comes from whether we value the in-file developer/LLM-context signal or not.

## Recommendation
**Standards win.** Keep ai-breadcrumbs.

Reasoning:
1. The validator's stated reason ("not read by any crawler") is technically correct but misses the audience. ai-breadcrumbs are not for crawlers; they are for **the humans and LLMs reading the raw HTML** (developers debugging, Claude sessions doing content work, external LLM consumers processing the HTML source). Those audiences exist, they DO see the comments, and the comments materially help them.
2. Standards docs invested heavily in this pattern (full spec in AI_BREADCRUMBS_SPECIFICATION.md). Archiving that and retrofitting hundreds of pages for no crawler benefit is a lot of churn for a philosophical disagreement.
3. The validator was changed based on an argument that isn't actually grounded in user need. This is exactly the kind of "validator grew from a single thread's decision" drift that the spec effort is trying to catch.
4. Countervailing argument: if this is your project, and you've looked at ai-breadcrumbs and think "yeah, this was a cargo-cult habit and I'd rather not keep maintaining them," then validator wins. That's a judgment call only the owner can make.

**Action if you accept the recommendation:**
- Delete lines 753-760 of `admin/validate-port-page-v2.js` (and any equivalent check in ship/venue validators). ~5 minute change.
- Update this rule: `decision: FINAL`, `provenance: V+S-agree` (since code+docs now align on keep), flip `check:` to the keep-version.
- Leave standards docs untouched.

**Action if you override toward validator:**
- Keep the validator check.
- Update this rule: `decision: FINAL`, `provenance: V+S-agree` (since code+docs now align on remove), flip `check:` to the remove-version, set `standards-backfill: yes`.
- Remove ai-breadcrumbs from AI_BREADCRUMBS_SPECIFICATION.md, ONBOARDING.md, skill-rules.json, etc. in Phase 6.
- Run a repo-wide strip of `<!-- ai-breadcrumbs...-->` blocks.
