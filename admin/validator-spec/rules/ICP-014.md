## USER DECISION (2026-04-15)
**Accepted recommendation: exact-match wins. Revert port validator's 30%-overlap relaxation.**

CLAUDE.md, ONBOARDING.md, and the ship validator all require exact string match between WebPage JSON-LD `description` and `ai-summary` meta. The port validator was relaxed to 30% word overlap in ICP-2 v2.1 without documentation; user accepts that the relaxation was a single-thread drift, not a considered policy change.

Implementation follow-up: revert `admin/validate-port-page-v2.js` lines 847-872 to string-equality check (matching ship validator lines 603-606). Small code change (~10 lines deleted, 2 added). Audit expected fallout: pages that currently pass via relaxed check but fail exact match. Fix: copy `ai-summary` content into `description` field on affected pages.

**After the validator update**, this rule transitions to `provenance: V+S-agree`. Until then, keep `V-S-conflict` to preserve history.

## Rule
A page's WebPage JSON-LD `description` must relate to the `ai-summary` meta — but how strictly is contested:
- **Port validator (ICP-2 v2.1):** 30% significant-word overlap is enough. Exact match not required.
- **Ship validator:** exact string match required.
- **CLAUDE.md / ONBOARDING.md (standards):** exact string match required.

Three sources, two positions. The validators don't even agree with each other.

## Why (rationale)
When ICP-2 v2.1 was introduced, the port validator was updated to accept "consistent" descriptions (relaxed) on the reasoning that `description` should be search-engine-oriented prose and `ai-summary` should be AI-dense fact-style; requiring them character-identical forced one voice to serve both audiences. The ship validator was not updated. The standards docs were not updated. Now three positions coexist in code and docs.

## Pass example (under port validator / relaxed)
```html
<meta name="ai-summary" content="Nassau is a tender-free port five minutes walk from Bay Street. Cable Beach jitney $1.25; Atlantis taxi $8."/>
<script type="application/ld+json">
{ "@type": "WebPage",
  "description": "Plan your Nassau cruise day — walk from the wharf to Bay Street in five minutes, take a jitney to Cable Beach, or taxi to Atlantis."
}
</script>
```
30%+ word overlap ("Nassau", "Bay", "Street", "Cable", "Beach", "Atlantis", "jitney", "taxi", "walk") — port validator passes.

## Pass example (under ship validator / exact)
Both strings identical character-for-character.

## Fail example (under both)
```html
<meta name="ai-summary" content="Nassau is a tender-free port..."/>
<script type="application/ld+json">
{ "@type": "WebPage", "description": "Explore beautiful beaches in the Caribbean." }
</script>
```
Port validator: too-low word overlap. Ship validator: strings don't match. Both fail.

## Implications

### If the relaxed (port) version wins
- **Validator change:** update `validate-ship-page.js` lines 603-606 to use the same 30%-overlap algorithm. Moderate — maybe 30 lines. Port's algorithm can be extracted into a shared utility.
- **Standards change:** CLAUDE.md and ONBOARDING.md text "EXACT MATCH" → "consistent (≥30% significant-word overlap)". Verbal update, no schema change.
- **Content cost:** zero. Pages that currently pass because they have an exact match continue to pass. Pages previously failing because they had divergent text may now pass.
- **Writer flexibility:** writers can tune `description` for search-engine click appeal without being forced to copy `ai-summary` verbatim.
- **Loss:** the clean "mirror exactly" rule is gone; replaced by a fuzzy rule that's harder to explain in one sentence.

### If the exact-match (ship + docs) version wins
- **Validator change:** revert port validator lines 847-872 back to string-equality. Trivial — maybe 10 lines deleted and 2 added.
- **Standards change:** none (already say "EXACT MATCH").
- **Content cost:** unknown without audit. Any port page that currently uses a prose `description` differing from dense `ai-summary` starts failing. Fixing: copy `ai-summary` into `description`.
- **Writer flexibility:** loses the ability to have a search-engine-oriented description distinct from the AI-oriented summary. One-size-fits-both.
- **Value:** easier to explain, easier to enforce, easier to mechanically generate from a single source of truth.

## Recommendation
**Exact-match wins.** Revert the port validator's relaxed check.

Reasoning:
1. Two validators + three docs all point to exact match originally. The port validator is the outlier; it was relaxed in what looks like a single-thread decision without documentation of why.
2. The practical argument for relaxation — "description serves search engines; ai-summary serves AI" — assumes they have genuinely different content goals. In practice on this site they do not. Both surfaces want "what is this page about in a standalone sentence or two." If that's true, one string can serve both.
3. The 30%-overlap algorithm is fuzzy. Fuzziness produces false negatives (a page rewrite shifts word overlap below threshold) and false positives (two unrelated summaries that happen to share common cruise-port vocabulary at >30%). Exact match has no such ambiguity.
4. "Exact match" is easier to explain to a writer ("copy-paste the ai-summary into description") than "30% significant-word overlap" ("… what's a significant word?").
5. Counter-argument: if there is a real use case for having distinct description vs ai-summary text (search-result snippet tuning), that use case wins. I don't see evidence that it's exercised today.
