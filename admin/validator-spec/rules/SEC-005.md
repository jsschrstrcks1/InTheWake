## Rule
Source files must not contain hardcoded API keys, tokens, passwords, or secrets. The validator pattern-matches against `password`, `secret`, common key prefixes (AKIA for AWS, AIzaSy for Google), and long base-64-like strings assigned to suspicious names.

## Why (rationale)
Every hardcoded secret on GitHub becomes public the moment it's pushed, and secret-scanning bots find them within seconds. Rotation becomes mandatory. Worse: if the key controls a paid API, the bill is the attacker's. Worst: if the key accesses user data or email, the breach is real.

The site's architecture is static-HTML-first; no server-side secret should ever appear in client JS. Admin tooling that calls external APIs for batch processing should use environment variables (`process.env.*`) not hardcoded values.

## Pass example
```js
// client-side JS — no secret needed, uses public analytics key only
const GA_ID = 'G-XXXXXXXX';  // designed to be public
```

```js
// admin/cjs — env var, never committed
const apiKey = process.env.SOME_API_KEY;
if (!apiKey) throw new Error('SOME_API_KEY not set');
```

## Fail example
```js
const password = "hunter2";  // BLOCKING
const apiSecret = "[AWS-ACCESS-KEY-PLACEHOLDER]";  // BLOCKING — real pattern: ^AKIA[0-9A-Z]{16}$
const googleMapsKey = "[GOOGLE-API-KEY-PLACEHOLDER]";  // BLOCKING — real pattern: ^AIza[0-9A-Za-z_-]{35}$
```
Validator emits: `No hardcoded API keys or secrets`.

> Note: the example uses non-regex-matching placeholders by design — a worked example that matches its own pattern would re-trigger any downstream credential scanner (including this project's `opensource-sanitizer` v1.2.0). The pattern definitions above (`^AKIA[0-9A-Z]{16}$`, `^AIza[0-9A-Za-z_-]{35}$`) are the source of truth for what this rule blocks.

## Fix guidance
- Rotate the exposed secret immediately — assume compromised.
- Move to environment variables for admin tooling.
- Public-by-design keys (GA4 measurement ID, published CDN URLs) are fine — the rule catches heuristics but the point is genuine secrets. A false-positive on a legitimately-public identifier is worth explaining in a `// SEC-005 reviewed: public analytics id` comment.

## Related incident prevention
This rule exists because the ingestion cost of a leaked key is measured in hours-to-notice-and-rotate versus the cost of blocking a commit in seconds. Err toward the block.
