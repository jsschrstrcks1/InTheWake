## Rule
JavaScript source must not call `document.write(...)`. Pattern match on `document\.write\(`.

## Why (rationale)
`document.write()` called after page load replaces the entire document — literally wipes everything. Called during load, it blocks the parser (no streaming render), tanks performance, and interacts unpredictably with preloaded resources and service workers. Browsers have been warning against it for over a decade; Chrome logs an intervention warning and may block it entirely during slow loads. There is no legitimate modern use.

The site uses a service worker (PWA-001) and precache manifest — `document.write` would break both.

## Pass example
```js
// Build DOM via createElement or innerHTML (SEC-004) instead
const notice = document.createElement('div');
notice.className = 'alert';
notice.textContent = 'Cached offline';
document.body.appendChild(notice);
```

## Fail example
```js
document.write('<div class="alert">Cached offline</div>'); // BLOCKING
```
Validator emits: `document.write() avoidance — can overwrite entire document`.

## Fix guidance
Use DOM APIs. If the code is creating HTML from a string, see SEC-004 for the innerHTML/textContent guidance. If it's legacy inline-script boilerplate (old analytics snippets sometimes used document.write), replace with the modern async-script version from the vendor's current docs.
