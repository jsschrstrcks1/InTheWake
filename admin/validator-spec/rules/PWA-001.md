## Rule
Whenever a release lands that changes any precached asset (HTML, CSS, JS, JSON data), `sw.js` must bump its `CACHE_VERSION` constant (or equivalent) and the precache manifest must reference assets with an updated `?v=` query string. Without this, returning visitors get stale cached content indefinitely.

## Why (rationale)
The site is a PWA. Service worker caches are aggressive by design — cache-first for images, network-first for HTML but cache as fallback. If the version constant doesn't change, the SW has no signal to invalidate. A reader who visited in March can load the March build forever. Standards describe this; no CI check enforces it.

## Pass example
```js
// sw.js at release v14.2.0
const CACHE_VERSION = 'v14.2.0-2026-04-15';
const PRECACHE = 'precache-' + CACHE_VERSION;
```
Paired with:
```html
<link rel="stylesheet" href="/assets/styles.css?v=3.010.400"/>
```

## Fail example
A release changes `assets/styles.css` but `sw.js` still says `CACHE_VERSION = 'v14.1.0-2026-03-01'`. Returning visitors keep using March styles until they hard-reload.

## Fix guidance
Wire a pre-release check: inspect `git log --oneline sw.js` and any file under `assets/` — if assets changed more recently than sw.js, fail the build. Phase 7 adds this to CI.

Manual workaround until then: every release that touches precached assets MUST bump `CACHE_VERSION` as part of the same commit. Document the bump in the release notes.
