## Rule
For each image file in a port's directory, the validator computes an MD5 hash and compares against `PLACEHOLDER_HASHES` — a constant set of hashes of known placeholder images (blank gradients, fake text rectangles, "Cordelia Empress Food Court" budget-line stand-in). Any match fails as BLOCKING.

## Why (rationale)
**The college-fjord escape.** Six `college-fjord-*.webp` files were blue-gradient rectangles with text rendered on them, pretending to be glacier photographs. They were shipped because no validator checked what was actually inside the bytes — only the filename and attribution JSON. MD5-matching against known placeholders is a cheap, deterministic way to catch byte-level fakes.

`PLACEHOLDER_HASHES` grows every time a new placeholder is discovered. As of April 2026 it includes the college-fjord gradients, "Cordelia_Empress_Food_Court" stand-ins, and several others. The set is maintained in the validator file itself at the top.

## Pass example
All 15 images in `ports/img/nassau/` have hashes absent from PLACEHOLDER_HASHES. Passes.

## Fail example
An image copied in to fill gallery slots — its MD5 matches a known placeholder. Validator emits: `Placeholder image detected: <filename>`. Validation blocks commit until the image is replaced or deleted.

## Fix guidance
Delete the placeholder. Source a real image of the port following the Image Verification Protocol (open with Read tool, confirm content matches caption). Do NOT simply generate a new placeholder and substitute it — the point of the protocol is that placeholders lie to readers.

## Extension note
When a new class of placeholder is discovered in the wild, add its MD5 to `PLACEHOLDER_HASHES`. The 2026-04 `college-fjord` incident was caught by a human auditor using Read; the hashes went into the set afterwards. This rule is effectively "all the lies we've caught so far" — it grows.
