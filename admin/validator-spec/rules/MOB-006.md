## Rule
No element should produce horizontal scroll at narrow viewports. The validator looks for inline-style patterns known to cause overflow (fixed wide pixel widths without containment, absolute positioning with overflow-triggering offsets, flex children without `min-width: 0`) and emits a BLOCKING error when found.

Overlaps with MOB-002 (inline width > 480px) — MOB-002 flags single elements; MOB-006 is the broader horizontal-scroll-root-cause check.

## Why (rationale)
Horizontal scroll on a page that shouldn't have any (text-only or mostly-text content) is the worst class of mobile bug: users swipe and watch content slide sideways instead of down. They can't read what's beyond the viewport without scrolling deliberately. Blocking severity reflects how visible and how-bad the bug is.

## Pass example
Page fits within the 375px viewport without horizontal scroll. All wide elements either constrain themselves (max-width: 100%) or wrap in overflow containers (MOB-004).

## Fail example
```html
<div style="position: absolute; left: 0; right: -100px;">...</div>
<!-- right: -100px pushes the element off-viewport-right; body scrolls sideways -->
```
Validator emits: `1 element(s) may cause horizontal scroll: <div style="... right: -100px">`.

## Fix guidance
Heuristics false-positive sometimes — the validator can't evaluate computed width without a browser. Real confirmation requires opening the page at 375px width in DevTools. When the validator flags something, investigate with DevTools; if it's a false positive, the detection is still cheap; if it's real, the fix is usually removing the absolute positioning or adding overflow containment.
