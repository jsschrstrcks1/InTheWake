## Rule
JavaScript source files must not call `eval()`. The check is a direct pattern match on `\beval\(`. Matches include `eval(userInput)`, `window.eval("...")`, and any other direct invocation.

## Why (rationale)
`eval()` executes arbitrary code in the current scope with full privileges. When combined with any user-originated string (a URL parameter, form input, JSON from an API), it becomes an immediate XSS / code-execution vector. There is no legitimate production use on this site — the site is static content with small interactive tools that use JSON parsing, not runtime code generation.

## Pass example
```js
const config = JSON.parse(configString);
const handler = { run: () => doWork() };
```

## Fail example
```js
const config = eval(configString); // BLOCKING
const result = eval("1 + " + userValue); // BLOCKING
```
Validator emits: `eval() — security risk`.

## Fix guidance
Use `JSON.parse` for JSON strings, `new Function()` (also forbidden) if you think you need dynamic code (you don't — rewrite as a dispatch table), or a prebuilt switch/map. If the code was inherited and you can't immediately rewrite, the rule stays blocking — no grandfather clause.
